## 1. ที่มาและวัตถุประสงค์

### 1.1 ที่มาและปัญหา (Background & Problem Statement)
ปัจจุบันธุรกิจร้านซักรีดขนาดย่อมมักประสบปัญหาจากการบริหารจัดการด้วยมือ (Manual Process) 
* **ความผิดพลาดของข้อมูล:** การจดบันทึกใส่สมุดทำให้เกิดความผิดพลาดในการคำนวณราคาและสรุปยอด
* **การสูญหายของพัสดุ:** ไม่มีระบบติดตามสถานะผ้าที่ชัดเจน ทำให้เกิดกรณีผ้าหายหรือสลับเจ้าของ
* **การบริการล่าช้า:** การค้นหาข้อมูลลูกค้าและประวัติเก่าทำได้ยากและใช้เวลานาน
* **กลุ่มผู้ได้รับผลกระทบ:** เจ้าของร้าน, พนักงานต้อนรับ, และลูกค้าที่ใช้บริการ

### 1.2 วัตถุประสงค์ของงาน (Objectives)
1.  เพื่อพัฒนาระบบ Web Application สำหรับบริหารจัดการงานรับ-ส่งผ้าและคำนวณค่าบริการ
2.  เพื่อลดข้อผิดพลาดในการปฏิบัติงานและป้องกันการสูญหายของเสื้อผ้าด้วยระบบ QR Code
3.  เพื่ออำนวยความสะดวกให้ลูกค้าสามารถตรวจสอบสถานะและประวัติการใช้บริการได้ด้วยตนเอง
4.  เพื่อสรุปรายงานรายรับ-รายจ่าย และข้อมูลการดำเนินงานให้ผู้บริหารทราบได้ทันที

## 2. ขอบเขตงาน (Scope of Work)

### 2.1 ในขอบเขต (In-Scope)
ระบบจะครอบคลุมฟังก์ชันการทำงานหลัก ดังนี้:
1.  **ระบบจัดการลูกค้า:** สมัครสมาชิก, บันทึกข้อมูลที่อยู่, และระบบสมาชิก VIP
2.  **ระบบรับงาน (Order):** สร้างใบรับงาน, คำนวณราคาอัตโนมัติ, และพิมพ์ใบรับงานพร้อม QR Code
3.  **ระบบติดตามสถานะ:** อัปเดตสถานะงาน (รับผ้า -> กำลังซัก -> เสร็จสิ้น) และตรวจสอบประวัติย้อนหลัง
4.  **ระบบค้นหาข้อมูล:** ค้นหาลูกค้าและออเดอร์จาก ชื่อ, เบอร์โทร, หรือรหัสใบงาน
5.  **ระบบรายงาน (Dashboard):** แสดงยอดรับงานประจำวัน, งานค้างส่ง, และรายได้รวม

### 2.2 นอกขอบเขต (Out-of-Scope)
เพื่อให้โครงการเสร็จทันตามกำหนดการ จึงขอตัดส่วนงานที่ไม่จำเป็นเร่งด่วนออก ดังนี้:
1.  การพัฒนา Mobile Application (Native App) บน iOS และ Android
2.  การเชื่อมต่อกับอุปกรณ์ฮาร์ดแวร์ POS หรือลิ้นชักเก็บเงินอัตโนมัติ
3.  ระบบติดตามรถขนส่งแบบ Real-time GPS (ใช้ระบบอัปเดตสถานะแทน)
4.  ระบบคำนวณเงินเดือนพนักงาน (Payroll) และระบบบัญชีภาษีเต็มรูปแบบ

### 2.3 สมมติฐาน (Assumptions)
* ผู้ใช้งาน (พนักงานและลูกค้า) มีอุปกรณ์สมาร์ทโฟนหรือแท็บเล็ตที่เชื่อมต่ออินเทอร์เน็ตได้
* ทางร้านมีเครื่องพิมพ์ (Printer) ทั่วไปสำหรับพิมพ์ใบรับงานกระดาษ A4 หรือ A5

### 2.4 ข้อจำกัด (Constraints)
* **ด้านเวลา:** ต้องดำเนินการให้แล้วเสร็จภายในภาคการศึกษานี้ (ก่อนวันที่ XX มีนาคม 2569)
* **ด้านเทคโนโลยี:** ระบบทำงานบน Web Browser เท่านั้น (Google Chrome, Safari, Edge)
* **ด้านกฎหมาย:** การเก็บข้อมูลส่วนบุคคลต้องมีระบบขอความยินยอมตาม พ.ร.บ. คุ้มครองข้อมูลส่วนบุคคล (PDPA)

## 4. เกณฑ์การตรวจรับงาน (Acceptance Criteria)

* **AC1:** ส่งมอบผลงานครบถ้วนตามรายการ D1 - D5 ภายในเวลาที่กำหนด
* **AC2:** ระบบสามารถทำงานได้จริงบน Web Browser โดยไม่มี Error ร้ายแรง (Critical Bug) ที่ทำให้ระบบหยุดทำงาน
* **AC3:** ฟังก์ชันหลัก (รับผ้า, เปลี่ยนสถานะ, ค้นหา, ดูรายงาน) ทำงานได้ถูกต้องตาม SRS
* **AC4:** ระบบ QR Code สามารถสแกนเพื่อดูสถานะงานได้จริงจากมือถือ
* **AC5:** ผ่านการทดสอบจากผู้ใช้งาน (UAT) โดยมีความพึงพอใจในระดับ "ผ่าน" ขึ้นไป

## 6. บทบาทและความรับผิดชอบ (Roles & Responsibilities)

### 6.1 ผู้เกี่ยวข้องหลัก
* **อาจารย์ที่ปรึกษา:** ผู้ให้คำปรึกษา, ผู้อนุมัติขอบเขต, และผู้ประเมินผลงาน
* **Project Manager (PM):** [ชื่อนักศึกษา A] - บริหารโครงการ, ติดตามงาน, ประสานงาน
* **Developer / System Analyst:** [ชื่อนักศึกษา B] - ออกแบบระบบ, พัฒนาโปรแกรม (Backend/DB)
* **UX/UI Designer / Frontend:** [ชื่อนักศึกษา C] - ออกแบบหน้าจอ, พัฒนาหน้าเว็บ, จัดทำคู่มือ

### 6.2 ตารางความรับผิดชอบ (RACI)

| กิจกรรม | PM | Dev | Design | อาจารย์ |
|:---|:---:|:---:|:---:|:---:|
| กำหนดขอบเขตโครงการ | **R/A** | C | C | **I/A** |
| จัดทำเอกสาร SRS / SoW | **R** | C | C | I |
| ออกแบบหน้าจอ (Mockup) | I | C | **R** | I |
| พัฒนา Coding / Database | I | **R** | C | I |
| ทดสอบระบบ (Testing) | A | R | R | I |
| นำเสนอและส่งมอบงาน | **R** | C | C | **A** |

*(R=ผู้ทำ, A=ผู้อนุมัติ/รับผิดชอบหลัก, C=ผู้ให้คำปรึกษา, I=ผู้รับทราบข้อมูล)*

## 8. การควบคุมการเปลี่ยนแปลง (Change Control)

หากต้องการเปลี่ยนแปลงขอบเขตงาน (Scope Creep) ที่ส่งผลกระทบต่อเวลาส่งมอบ:
1.  ผู้เสนอการเปลี่ยนแปลงต้องแจ้งให้ PM ทราบพร้อมเหตุผล
2.  ทีมงานประชุมเพื่อประเมินผลกระทบ (Impact Analysis)
3.  หากเป็นการเปลี่ยนแปลงใหญ่ ต้องได้รับความเห็นชอบจากอาจารย์ที่ปรึกษาก่อนลงมือทำ
4.  บันทึกการเปลี่ยนแปลงลงในเอกสาร "บันทึกการแก้ไข" เสมอ

## 9. คุณภาพและการทดสอบ (Quality & Testing)

* **Unit Testing:** ผู้พัฒนาทดสอบฟังก์ชันย่อยของตนเองทันทีหลังเขียนโค้ดเสร็จ
* **Integration Testing:** ทดสอบการทำงานร่วมกันระหว่างหน้าเว็บและฐานข้อมูล
* **User Acceptance Testing (UAT):** ให้เพื่อนกลุ่มอื่นหรือผู้ใช้งานจำลอง ลองใช้งานระบบตาม Scenario ที่กำหนดเพื่อหาจุดบกพร่อง (Bugs) ก่อนส่งมอบจริง

---

## 10. การส่งมอบและปิดงาน (Handover)

เมื่อสิ้นสุดโครงการ สิ่งที่ต้องส่งมอบประกอบด้วย:
1.  **Source Code:** อัปโหลดขึ้น GitHub หรือส่งเป็นไฟล์ Zip
2.  **Database Export:** ไฟล์ .sql สำหรับติดตั้งฐานข้อมูล
3.  **เล่มรายงานฉบับสมบูรณ์:** รวม SRS, Design, และผลการทดสอบ
4.  **VDO Presentation:** คลิปแนะนำการใช้งานระบบความยาว 3-5 นาที
